## 10. Session 2026-06-15 (later): parallel-pipeline bug found + fixed — §9 misdiagnosis OVERTURNED

**Trigger:** "all jobs finished correctly?" → verified the §9 reruns landed, then

### 10.1 The §9 conclusion was wrong

§9 (line ~753–755) and §8.6 item 1(a) attribute the parallel 0% to the model.
Root-caused to the **`process_parallel` design**, not the models:

- **Every one of the 7 models scores _exactly_ 0.0% (0/200) on `parallel`** — Gemma-3-1b,
  Llama-3.2-{1B,3B}, Phi-4-mini, Qwen2.5-{0.5B,1.5B,3B}. Exact-zero with no spread
  (while `multiple` shows real spread: Llama-3B 57.5%, Gemma-1b 44%) is the signature
  of a code bug, not a capability pattern.
- **Layers ruled out:** routing is correct (`parallel` ∈ `_PARALLEL_CATEGORIES` →
  `process_parallel` → `format_parallel_results`); serialization is correct (renders
  all calls, not `results[0]`). The "1/200 multi-call" in an early grep was a **regex
  false positive** (a `(AA)` paren inside a string arg) — all 200 are single-call.
- **Root cause (two compounding bugs in `src/vllm_backend.py:process_parallel`):**
  1. `"maxItems": min(len(fn_names), 5)` capped the call list by the count of
     *distinct* available functions. BFCL parallel often needs the **same** function
     called N times (e.g. `parallel_0` = `spotify.play` twice, different args); when one
     function is available, `maxItems = min(1,5) = 1` → the grammar *forbids* >1 call →
     deterministic `"Wrong number of functions."` for all models.
  2. Args were extracted once per *distinct* name from the whole prompt, so repeated
     calls could never get different arguments.
- Ground-truth call-count distribution for `parallel`: `{2:109, 3:52, 4:36, 6:1, 8:2}`
  — **no item needs 1 call**, so a single-call pipeline scores 0 by construction.

### 10.2 Fix (uncommitted)

- **`src/vllm_backend.py`** — `process_parallel` rewritten to **one** guided generation
  of `{"calls": [{"name", "arguments"}, ...]}`, `name` allowed to repeat, each element
  carrying its own args. Array items are a `oneOf` over per-function call objects
  (`name` pinned with `const` + that function's `_build_args_json_schema`).
  `minItems:1, maxItems:8` (covers the distribution; no distinct-count cap). Factored
  out `_coerce_args` (shared) and added `_parse_parallel_calls` (tolerates extra text
  on the no-guided path).
- **`src/prompt.py`** — `build_parallel_selection_prompt` now asks for the
  `{name, arguments}` format and explicitly to repeat a function with different args.
- **`tests/test_parallel_pipeline.py`** (new) — 4 regression tests (schema not capped to
  1; same function twice with distinct args; no-guided parse amid extra text;
  empty-calls fallback). Full suite **78 passed**.
- **`scripts/smoke_parallel_schema.py`** (new) — offline de-risk: both xgrammar (vLLM
  default) and outlines compile the `oneOf`+`const` schema. Both pass — no GPU needed.
