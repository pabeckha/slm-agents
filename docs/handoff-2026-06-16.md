## 1. Headline: the `parallel` 0% is a documented design limit, not a bug — reframe, don't retract

Tracing the last finished job (28668050, Qwen-7B `parallel_multiple` 72.5%)
surfaced that the BFCL `parallel` 0% baked into the thesis is the result of the
**old, deliberately documented** pipeline design, lifted by a later extension.

- **Mechanism:** methodology §387–401 documents the parallel stage as emitting one
  argument object per *distinct* function name (call array `maxItems = number of
  distinct candidates`), with the rationale that deterministic temp-0 extraction
  gives identical args for repeated calls. Every BFCL `parallel` case has exactly
  one distinct function → one call → 0% by construction.
- **So it is a valid result for the architecture as evaluated, NOT a measurement
  bug / artifact.** Commit `2faa0df` ("joint guided multi-call generation",
  `_build_parallel_calls_schema`) **extends** the architecture (array of complete
  call objects generated jointly, so the same function repeats with distinct args)
  — exactly the remedy the discussion already named.
- **Post-fix proof (plain CD, Qwen, 200 cases):** `parallel` 0.5/1.5/3/7B =
  **9.5 / 64.0 / 74.5 / 79.0%**; 7B `parallel_multiple` **38.5% → 72.5%**.
  Not "solved" — 0.5B is still ~9.5%, so it stays a hard, size-sensitive category.
- This **corrects issue #173 / handoff §10**, which called it a "deterministic
  pipeline bug / artifact" — that overshoots. The thesis §5.3 already calls it
  *structural* (not "model weakness"); only the permanent "ceiling … is 0%"
  overclaim needs softening. Full framing + exact thesis edit locations:
  `docs/decisions/thesis-parallel-artifact-correction.md`.
